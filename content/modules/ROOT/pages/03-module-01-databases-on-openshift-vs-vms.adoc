= Module 1: Databases on OpenShift vs. VMs

include::partial$_attributes.adoc[]

DataStore Inc has run PostgreSQL on virtual machines for over a decade. The VMs are reliable but slow to provision, expensive to maintain, and require manual failover. Your CTO has asked you to evaluate OpenShift as the new home for PostgreSQL, but before you deploy anything, you need to understand how containerized databases differ from VM-based ones — and what that means for latency, consistency, and operations.

In this module, you will explore the OpenShift environment, examine how OpenShift manages persistent storage, and compare VM-based and containerized PostgreSQL to build the foundation for your migration plan.

== Learning objectives

By the end of this module, you will be able to:

* Explore the OpenShift storage landscape and identify StorageClasses suitable for database workloads
* Describe the key differences between VM-based and containerized PostgreSQL, including latency, I/O consistency, and operational trade-offs
* Identify the 4 core OpenShift primitives — StatefulSet, Service, Secret, and PodDisruptionBudget — that make databases production-ready on OpenShift
* Determine when running a database on OpenShift is the right architectural choice

== Exercise 2: VMs vs. containers — understanding the trade-offs

DataStore Inc's current PostgreSQL VM setup uses dedicated resources: 16 GB RAM, 8 vCPUs, and a 500 GB SSD, reserved exclusively for 1 database instance. You need to understand exactly what changes when that database moves to OpenShift, and where the trade-offs lie.

. Document the VM baseline configuration for reference. Create a ConfigMap that captures the current VM profile:
+
[source,bash]
----
cat <<'EOF' | oc apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
  name: vm-baseline-spec
  namespace: {user}-database
data:
  spec.txt: |
    DataStore Inc - VM-based PostgreSQL baseline
    =============================================
    Provisioning time : 2-3 weeks (VM request, storage, OS, PostgreSQL install, tuning)

    Resource allocation (dedicated, always reserved):
      CPU     : 8 vCPUs   (dedicated, no sharing — idle 60% of the time)
      Memory  : 16 GB RAM (dedicated, no sharing)
      Storage : 500 GB SSD (dedicated LUN, fixed size)

    High availability:
      Standby VM configured manually
      Cron-monitored replication
      Manual failover: 15-30 min RTO, DBA on-call required

    Backups:
      pg_dump via cron
      rsync to NFS share
      No coordination with storage snapshots
EOF
----
+
[source,text]
----

. Now create a ConfigMap documenting the latency, consistency, and decision criteria — the 3 areas DataStore Inc's leadership asked you to evaluate:
+
[source,bash]
----
cat <<'EOF' | oc apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
  name: migration-evaluation
  namespace: {user}-database
data:
  latency.txt: |
    LATENCY CONSIDERATIONS
    ======================
    VM-based:
    - Direct hardware access for most storage configurations
    - Network latency predictable on dedicated storage VLANs
    - Typical SSD write latency: 0.1-0.5 ms

    OpenShift (container with CSI block storage):
    - CSI driver adds a thin layer between pod and storage (gp3, ODF RBD, etc.)
    - Typical overhead vs bare metal: 2-5% for block storage
    - PostgreSQL WAL writes are the critical path — sync I/O latency matters most
    - Mitigation: use local NVMe StorageClass for latency-sensitive workloads

    Verdict for DataStore Inc:
    - Retail analytics (OLAP queries) : OpenShift overhead acceptable (<5% on reads)
    - High-frequency OLTP (<1ms P99) : evaluate local NVMe StorageClass first

  consistency.txt: |
    CONSISTENCY CONSIDERATIONS
    ==========================
    VM-based:
    - No container restart risk during transactions (OS-level stability)
    - Network partitions are rare in dedicated VM storage networks

    OpenShift (container):
    - Pod evictions can interrupt long-running transactions
    - Mitigation: set terminationGracePeriodSeconds >= 60 to allow PostgreSQL
      to write its checkpoint cleanly before the pod is killed
    - Multi-replica setups need Patroni or an operator for consistent primary election
    - ReadWriteOnce PVCs prevent split-brain: only 1 pod writes at a time

    Verdict for DataStore Inc:
    - Standard transactional workloads: OpenShift is safe with proper configuration
    - Critical recommendation: always set terminationGracePeriodSeconds: 60

  decision.txt: |
    WHEN OPENSHIFT IS THE RIGHT CHOICE
    ====================================
    - Workloads where 2-5% I/O overhead is acceptable
    - Teams that need developer self-service database provisioning
    - Applications requiring consistent environments across dev/staging/prod
    - Workloads where HA automation matters more than raw single-node throughput
    - Organizations migrating from VMs to reduce infrastructure cost and provisioning time

    WHEN TO CONSIDER ALTERNATIVES
    ==============================
    - Workloads requiring < 1 ms P99 storage I/O with no tolerance for overhead
    - Databases > 10 TB with specialized hardware requirements
    - Legacy databases with kernel module dependencies incompatible with containers
EOF
----

== Exercise 3: The 4 OpenShift primitives for database workloads

Before deploying PostgreSQL in Module 2, you need to understand the 4 core Kubernetes objects that make stateful databases production-ready on OpenShift. In this exercise, you will review example definitions for each one.

. Create a reference ConfigMap containing annotated examples of all 4 primitives:
+
[source,bash]
----
cat <<'EOF' | oc apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
  name: postgres-primitives-reference
  namespace: {user}-database
data:
  1-statefulset.yaml: |
    # StatefulSet: provides stable network identity and ordered pod management.
    # Unlike Deployments, StatefulSets guarantee pod ordering (pod-0 before pod-1)
    # and each pod gets its own PVC from volumeClaimTemplates.
    # terminationGracePeriodSeconds: 60 gives PostgreSQL time to checkpoint.
    apiVersion: apps/v1
    kind: StatefulSet
    metadata:
      name: postgres
    spec:
      serviceName: postgres
      replicas: 1
      selector:
        matchLabels:
          app: postgres
      template:
        spec:
          terminationGracePeriodSeconds: 60
          containers:
          - name: postgres
            image: registry.redhat.io/rhel9/postgresql-{postgresql_version}:latest
            resources:
              requests:
                cpu: 500m
                memory: 1Gi
              limits:
                cpu: 2000m
                memory: 4Gi
            volumeMounts:
            - name: data
              mountPath: /var/lib/pgsql/data
      volumeClaimTemplates:
      - metadata:
          name: data
        spec:
          accessModes: [ReadWriteOnce]
          storageClassName: gp3-csi-retain
          resources:
            requests:
              storage: 10Gi

  2-service.yaml: |
    # Service (headless): stable DNS name for each pod.
    # postgres-0.postgres.{namespace}.svc.cluster.local
    # Used internally for pod-to-pod replication in HA setups.
    apiVersion: v1
    kind: Service
    metadata:
      name: postgres
    spec:
      clusterIP: None
      selector:
        app: postgres
      ports:
      - port: 5432
        name: postgres

  3-secret.yaml: |
    # Secret: stores database credentials.
    # Never hardcode passwords in YAML or ConfigMaps.
    # OpenShift injects Secrets as environment variables or volume files.
    apiVersion: v1
    kind: Secret
    metadata:
      name: postgres-credentials
    type: Opaque
    stringData:
      POSTGRESQL_USER: datastore_app
      POSTGRESQL_PASSWORD: changeme-use-a-generated-password
      POSTGRESQL_DATABASE: datastore_prod

  4-pdb.yaml: |
    # PodDisruptionBudget: prevents OpenShift from evicting too many DB pods at once
    # during node maintenance, upgrades, or drains.
    # minAvailable: 1 means at least 1 replica must stay running.
    apiVersion: policy/v1
    kind: PodDisruptionBudget
    metadata:
      name: postgres-pdb
    spec:
      minAvailable: 1
      selector:
        matchLabels:
          app: postgres
EOF
----
+
[source,text]
----

----

== Troubleshooting

*Issue*: `oc login` fails with `x509: certificate signed by unknown authority`

*Solution*:

. Add `--insecure-skip-tls-verify` to the login command for the lab environment:
+
[WARNING]
====
`--insecure-skip-tls-verify` disables TLS certificate validation. This is only appropriate in lab environments with self-signed certificates. Never use this flag against production clusters.
====
+
[source,bash]
----
oc login {console_url} --username={user} --password={password} --insecure-skip-tls-verify
----

*Issue*: `oc get storageclass` returns no resources

*Solution*:

. StorageClasses are cluster-scoped — do not add `-n {user}-database`. Run `oc get storageclass` without a namespace flag.

*Issue*: `oc get csv` returns no postgres results

*Solution*:

. Search across all namespaces: `oc get csv -A | grep -i postgres`
. The Crunchy PostgreSQL operator may be installed cluster-wide in `openshift-operators`.

*Issue*: ConfigMap not found during cleanup

*Solution*:

. Run `oc get configmap` to list what exists, then delete only the ConfigMaps that are present. This error is expected if a step failed or was skipped.

== Module summary

You have completed DataStore Inc's pre-migration evaluation and built the foundation for everything that follows.

*What you accomplished:*

* Explored the OpenShift storage landscape and confirmed StorageClasses are available for database workloads
* Documented the architectural differences between VM-based and containerized PostgreSQL, including latency (2-5% CSI overhead) and consistency (terminationGracePeriodSeconds: 60 requirement)
* Reviewed the 4 core OpenShift primitives — StatefulSet, Service, Secret, and PodDisruptionBudget — that make PostgreSQL production-ready
* Confirmed that DataStore Inc's cluster has the worker node count and permissions needed for a successful migration

*Key takeaways:*

* PersistentVolumeClaims give containers durable storage that survives pod restarts — data is never lost on pod failure alone
* Use a StorageClass with `RECLAIMPOLICY: Retain` for databases to prevent accidental PVC deletion from destroying data
* StatefulSets are required for databases: they provide stable network identity, ordered startup, and per-pod PVCs that Deployments cannot
* Container I/O overhead is typically 2-5% — acceptable for most workloads in exchange for automated HA and self-service provisioning
* Always set `terminationGracePeriodSeconds: 60` — it gives PostgreSQL time to write its checkpoint cleanly before the pod is killed
