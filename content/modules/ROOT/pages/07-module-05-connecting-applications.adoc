= Module 5: Connecting applications to databases

include::partial$_attributes.adoc[]

DataStore Inc's retail analytics API team is ready. They have a containerized application that reads from the PostgreSQL database and serves inventory data to the retail stores. The CTO wants to see the full stack running end to end — application pods connecting to the database pod — before the production cutover.

In this module, you will deploy a stateless application Deployment that connects to the PostgreSQL instance from Module 2, inject credentials using the existing Secret, verify connectivity from multiple replicas, and manage the application lifecycle with rolling updates and graceful reconnection through database restarts.

== Learning objectives

By the end of this module, you will be able to:

* Explain the difference between stateless and stateful applications and why only stateless applications scale freely on OpenShift
* Deploy an application Deployment that connects to PostgreSQL using the Kubernetes Service DNS name and injected credentials
* Verify that all application replicas can reach the database and perform both read and write operations
* Scale the application independently of the database and trigger a zero-downtime rolling update
* Observe application reconnection behavior after a database pod restart

[#exercise-1-deploy-the-analytics-api-and-understand-stateless-vs-stateful-patterns]
== Exercise 1: Deploy the analytics API and understand stateless vs. stateful patterns

DataStore Inc's analytics API is stateless — it holds no data of its own. Every request reads from the database and returns a result. Any pod can handle any request, which means it can be scaled up or down freely. The PostgreSQL database is stateful — its data lives in a PVC that is tied to a specific pod. Scaling a stateful application requires coordination that a Deployment alone cannot provide.

. Review the architectural difference before deploying. Create a ConfigMap documenting how the 2 tiers interact:
+
[source,bash]
----
cat <<'EOF' | oc apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-architecture-reference
  namespace: {user}-database
data:
  stateless-app.txt: |
    STATELESS APPLICATION (DataStore Analytics API)
    ================================================
    - No per-pod state: any replica can handle any request
    - Connects to the database on every request, holds no data
    - Scales freely: oc scale deployment datastore-api --replicas=N
    - Deployed as a Deployment (not StatefulSet): pods are interchangeable
    - Rolling updates are zero-downtime: new pods start before old ones stop
    - Crash recovery: scheduler replaces pod anywhere on the cluster
    - PVC: none required
    - Disaster recovery: traffic redirection only. Point the load balancer at a
      second cluster running identical Deployment manifests and the app is recovered.
      RTO is seconds. No storage coordination required.

  stateful-app.txt: |
    STATEFUL APPLICATION (PostgreSQL database)
    ==========================================
    - Per-pod state: data lives in a PVC bound to a specific pod name
    - Pod identity matters: postgres-0 always mounts data-postgres-0
    - Cannot scale freely: replicas need replication and primary election
    - Deployed as a StatefulSet: pods have stable names and ordered startup
    - Updates require care: StatefulSet rolls one pod at a time
    - Crash recovery: same pod name reattaches to same PVC
    - PVC: required (10Gi gp3-csi, bound to postgres-0)
    - Storage type matters: gp3-csi is block storage -- single availability zone
      only. The database pod is zone-affined to the node where the PVC is bound.
      For multi-AZ resilience, use OpenShift Data Foundation (ODF), which provides
      3-way synchronous replication across availability zones.
    - Disaster recovery: state must be recovered, not just traffic redirected.
      Two approaches: (1) backup/restore -- restore from a VolumeSnapshot, RPO
      equals time since last snapshot, RTO equals restore + startup time;
      (2) storage replication -- ODF Regional-DR replicates the PVC to a second
      cluster asynchronously, RPO is near-zero, RTO is automated failover time.
EOF
----
+
[source,text]
----
configmap/app-architecture-reference created
----

[NOTE]
====
*Storage is the primary challenge when containerizing stateful workloads.* The `gp3-csi` StorageClass used in this workshop provisions AWS EBS block volumes, which are single-availability-zone resources. If the node hosting `postgres-0` is in `us-east-1a`, the PVC is also in `us-east-1a` -- the database pod cannot be rescheduled to a node in a different AZ without losing access to its data. In production, distribute database nodes across multiple AZs and use OpenShift Data Foundation (ODF), which replicates data 3 ways across AZs synchronously. ODF also enables resource consolidation: instead of provisioning a dedicated EBS volume per database, multiple databases share a storage pool with guaranteed isolation.
====

. Create a ConfigMap containing the analytics API startup script. This script polls the products table every 30 seconds, simulating the request loop of a real inventory API:
+
[source,bash]
----
cat <<'EOF' | oc apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
  name: datastore-api-scripts
  namespace: {user}-database
data:
  api.sh: |
    #!/bin/bash
    echo "DataStore Inc Analytics API v1 starting"
    echo "Database host : $PGHOST"
    echo "Database name : $PGDATABASE"
    echo "Database user : $PGUSER"

    until psql -c "SELECT 1;" >/dev/null 2>&1; do
      echo "$(date '+%H:%M:%S'): Waiting for database connection..."
      sleep 5
    done

    echo "$(date '+%H:%M:%S'): Connected. Starting inventory poll."

    while true; do
      COUNT=$(psql -t -c "SELECT COUNT(*) FROM products;" 2>/dev/null | tr -d ' ')
      if [ $? -eq 0 ]; then
        echo "$(date '+%H:%M:%S'): $COUNT products in inventory"
        psql -c "SELECT id, sku, name, price, stock FROM products ORDER BY id;"
      else
        echo "$(date '+%H:%M:%S'): Database connection lost. Retrying on next cycle."
      fi
      sleep 30
    done
EOF
----
+
[source,text]
----
configmap/datastore-api-scripts created
----

. Deploy the analytics API as a Deployment with 2 replicas. The Deployment uses the existing `postgres-credentials` Secret for database credentials and mounts the script ConfigMap:
+
[source,bash]
----
cat <<'EOF' | oc apply -f -
apiVersion: apps/v1
kind: Deployment
metadata:
  name: datastore-api
  namespace: {user}-database
  labels:
    app: datastore-api
    version: v1
spec:
  replicas: 2
  selector:
    matchLabels:
      app: datastore-api
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 0
      maxSurge: 1
  template:
    metadata:
      labels:
        app: datastore-api
        version: v1
    spec:
      containers:
      - name: api
        image: registry.redhat.io/rhel9/postgresql-{postgresql_version}:latest
        command: ["/bin/bash", "/scripts/api.sh"]
        env:
        - name: PGHOST
          value: "postgres"
        - name: PGUSER
          valueFrom:
            secretKeyRef:
              name: postgres-credentials
              key: POSTGRESQL_USER
        - name: PGPASSWORD
          valueFrom:
            secretKeyRef:
              name: postgres-credentials
              key: POSTGRESQL_PASSWORD
        - name: PGDATABASE
          valueFrom:
            secretKeyRef:
              name: postgres-credentials
              key: POSTGRESQL_DATABASE
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 500m
            memory: 256Mi
        volumeMounts:
        - name: scripts
          mountPath: /scripts
      volumes:
      - name: scripts
        configMap:
          name: datastore-api-scripts
          defaultMode: 0755
EOF
----
+
[source,text]
----
deployment.apps/datastore-api created
----
+
[NOTE]
====
`PGHOST: postgres` uses the short DNS form. Because the Deployment and the PostgreSQL Service are both in the `{user}-database` namespace, Kubernetes resolves `postgres` to the Service's endpoints without needing the fully qualified `postgres.{user}-database.svc.cluster.local` name. `maxUnavailable: 0` and `maxSurge: 1` ensure the rolling update never removes a pod before a replacement is ready.

The `until psql -c "SELECT 1;"` loop in `api.sh` is a simple database readiness gate. In production, use an `initContainers` entry instead -- it runs before the main container starts and exits with a non-zero code if the database is not yet ready, which prevents the application container from starting in an unusable state and keeps the pod in `Init:0/1` until the dependency is satisfied.

This Deployment uses the PostgreSQL image as a convenient base that ships with `psql` and `bash`. In a real environment, the analytics API would use a Python, Go, or Node.js application image. The image choice does not affect the connectivity, scaling, or rolling update patterns demonstrated in this module.
====

[NOTE]
====
The workshop environment pre-provisions a pull secret for `registry.redhat.io` in your namespace. If you run this exercise outside the workshop, create one first: `oc create secret docker-registry redhat-registry --docker-server=registry.redhat.io --docker-username=<RHN_USERNAME> --docker-password=<RHN_PASSWORD> -n {user}-database`, then link it: `oc secrets link default redhat-registry --for=pull -n {user}-database`.
====

  name: datastore-api-scripts

    #!/bin/bash
    echo "DataStore Inc Analytics API v2 starting"
    echo "Database host : $PGHOST"

    echo "Database user : $PGUSER"

    until psql -c "SELECT 1;" >/dev/null 2>&1; do
      echo "$(date '+%H:%M:%S'): Waiting for database connection..."
      sleep 5

    echo "$(date '+%H:%M:%S'): Connected. Starting inventory poll (v2)."

      RESULT=$(psql -t -c \
        "SELECT COUNT(*), COALESCE(SUM(price * stock), 0) FROM products;" \
        2>/dev/null)
      PSQL_EXIT=$?
      if [ $PSQL_EXIT -eq 0 ] && [ -n "$RESULT" ]; then

EOF

== Exercise 4: Cache inventory data with Red Hat Data Grid

DataStore Inc's analytics dashboard polls inventory totals every few seconds across 20 or more application pods. That volume of direct PostgreSQL queries causes measurable load. Red Hat Data Grid provides an in-memory cache layer between the application and the database. The API checks the cache first and queries PostgreSQL only on a cache miss, then writes the result back to the cache with a time-to-live. This is the cache-aside pattern.

=== Learning objectives

* Verify that the Red Hat Data Grid operator is installed and ready
* Deploy a single-replica Data Grid instance using the `Infinispan` custom resource
* Configure the analytics API to use the cache-aside pattern
* Observe cache hits and misses in the application logs
* Confirm the cached value via the Data Grid REST API

=== Step 7: Update the api.sh script to use the cache-aside pattern

Replace the ConfigMap with an updated `api.sh` that checks the cache on every poll:

[source,bash]
----
cat <<'SCRIPT' > /tmp/api-v3.sh
#!/bin/bash
set -euo pipefail

export PGPASSWORD="$POSTGRESQL_PASSWORD"
DG_BASE="https://${DG_HOST}:11222"

# Create the inventory cache on startup (409 if it already exists is acceptable)
curl -sk -u "$DG_USER:$DG_PASSWORD" -X POST \
  "$DG_BASE/rest/v2/caches/inventory" \
  -H "Content-Type: application/json" \
  -d '{"distributed-cache":{"mode":"SYNC"}}' >/dev/null 2>&1

while true; do
  HTTP_CODE=$(curl -sk -u "$DG_USER:$DG_PASSWORD" \
    -o /tmp/cache_val -w "%{http_code}" \
    "$DG_BASE/rest/v2/caches/inventory/total")
  CACHED_VAL=$(cat /tmp/cache_val)

  if [ "$HTTP_CODE" = "200" ] && [ -n "$CACHED_VAL" ]; then
    echo "$(date '+%H:%M:%S'): [CACHE HIT] $CACHED_VAL"
  else
    RESULT=$(psql -h "$POSTGRESQL_SERVICE_HOST" \
      -U "$POSTGRESQL_USER" \
      -d "$POSTGRESQL_DATABASE" \
      -t -c "SELECT COUNT(*), COALESCE(SUM(price * stock), 0) FROM products;" 2>/dev/null)
    PSQL_EXIT=$?
    if [ $PSQL_EXIT -eq 0 ] && [ -n "$RESULT" ]; then
      COUNT=$(echo "$RESULT" | cut -d'|' -f1 | tr -d ' ')
      VALUE=$(echo "$RESULT" | cut -d'|' -f2 | tr -d ' ')
      SUMMARY="${COUNT} products, \$${VALUE}"
      curl -sk -u "$DG_USER:$DG_PASSWORD" -X PUT \
        -H "Content-Type: text/plain" \
        -H "timeToLiveSeconds: 60" \
        "$DG_BASE/rest/v2/caches/inventory/total" \
        -d "$SUMMARY" >/dev/null 2>&1
      echo "$(date '+%H:%M:%S'): [CACHE MISS] $SUMMARY (cached for 60s)"
    else
      echo "$(date '+%H:%M:%S'): Database connection lost. Retrying."
    fi
  fi
  sleep 15
done
SCRIPT

oc create configmap datastore-api-scripts \
  --from-file=api.sh=/tmp/api-v3.sh \
  --dry-run=client -o yaml | oc apply -f - -n {user}-database

== Troubleshooting

*Issue*: Application pods start but log `Waiting for database connection...` indefinitely

*Solution*:

. Verify the Service resolves correctly from inside an API pod:
+
[source,bash]
----
oc exec -it $(oc get pod -l app=datastore-api -n {user}-database \
  -o jsonpath='{.items[0].metadata.name}') -n {user}-database -- \
  bash -c "getent hosts postgres"
----
. If DNS resolution fails, confirm the `postgres` Service exists in the same namespace:
+
[source,bash]
----
oc get service postgres -n {user}-database
----
. If the Service exists but the pod cannot connect, check whether the PostgreSQL pod is ready:
+
[source,bash]
----
oc get pod postgres-0 -n {user}-database
----

*Issue*: Application pods fail to start — `Error: failed to create containerd task`

*Solution*:

. Check whether the `datastore-api-scripts` ConfigMap exists and has the `api.sh` key:
+
[source,bash]
----
oc get configmap datastore-api-scripts -n {user}-database -o jsonpath='{.data.api\.sh}' | head -5
----
. If the ConfigMap is missing or empty, recreate it from the step in Exercise 1.

*Issue*: `oc rollout undo` reverts the pod template but the v1 script does not appear

*Solution*:

. `oc rollout undo` reverts the pod template (including the `kubectl.kubernetes.io/last-applied-configuration` annotation) but does not touch ConfigMaps. Manually reapply the v1 ConfigMap from Exercise 1 to restore the original script.

*Issue*: `INSERT INTO products` fails with `ERROR: permission denied`

*Solution*:

. The `datastore_app` user was created with full rights to the `datastore_prod` database but may not have INSERT rights if the table was created by a different superuser. Grant rights explicitly:
+
[source,bash]
----
oc exec postgres-0 -n {user}-database -- \
  psql -U postgres -d datastore_prod \
  -c "GRANT INSERT, SELECT, UPDATE ON products TO datastore_app;"
----

== Module summary

DataStore Inc's full stack is now running on OpenShift: a stateless analytics API Deployment connected to the stateful PostgreSQL StatefulSet, both in the same namespace, with credentials injected from a shared Secret.

The work across all 5 modules illustrates the two tiers of disaster recovery. The analytics API has trivially simple DR: redeploy the Deployment manifests on any cluster and point traffic at it. The PostgreSQL database requires explicit DR planning. The VolumeSnapshot taken in Module 3 defines a concrete Recovery Point Objective (RPO) for DataStore Inc: any data written after that snapshot would be lost in a full-cluster failure. The restore performed in Module 4 demonstrated the Recovery Time Objective (RTO): the time from issuing the restore command to a verified, running database. For lower RPO, ODF Regional-DR replicates PVCs to a second cluster asynchronously on a configurable schedule. For zero RPO, ODF Metro-DR uses synchronous replication between datacenters close enough to support it.

*What you accomplished:*

* Deployed a stateless analytics API Deployment with 2 replicas that connects to PostgreSQL using the `postgres` Service DNS name and credentials from the existing `postgres-credentials` Secret
* Verified DNS resolution and database connectivity from both application replicas, and confirmed the read and write paths work end to end
* Scaled the Deployment to 4 replicas with a single command — no database configuration change required
* Deployed API v2 with a rolling update using `maxUnavailable: 0` for zero-downtime deployment, and observed the application automatically reconnect after a database pod restart
* Rolled back to v1 using `oc rollout undo` and confirmed rollout history tracking

*Key takeaways:*

* Stateless applications can scale freely because any pod can handle any request — they hold no per-pod state. Stateful applications cannot scale the same way because data and identity are tied to a specific pod. This distinction drives the choice between `Deployment` and `StatefulSet`
* Disaster recovery complexity follows the stateless/stateful boundary. Recovering a stateless tier is traffic redirection — point the load balancer at a second cluster and the application is running in seconds. Recovering a stateful tier requires storage recovery: either restoring from a VolumeSnapshot (backup/restore, RPO = age of last snapshot) or failing over replicated PVCs (ODF Regional-DR for async replication, ODF Metro-DR for synchronous zero-RPO replication between nearby datacenters)
* Storage is the primary challenge when containerizing stateful workloads. Block storage (gp3-csi) is single-AZ — the database pod is zone-affined to the node where the PVC lives. OpenShift Data Foundation (ODF) removes this constraint with 3-way synchronous replication across availability zones
* The short DNS form `postgres` resolves to the Service in the same namespace — use the fully qualified name `postgres.{user}-database.svc.cluster.local` when connecting from a different namespace
* ConfigMap changes do not restart pods automatically — always trigger a `rollout restart` or patch the pod template to deploy a configuration change
* `maxUnavailable: 0` combined with `maxSurge: 1` guarantees zero-downtime rolling updates: a new pod is fully ready before any old pod is removed. This is the correct setting for database-connected applications where reconnection during startup has a cost
* Application-level reconnection logic is required for production. The poll loop in this module demonstrates the pattern. For production workloads, use a connection pool such as pgBouncer in front of PostgreSQL: it maintains a warm pool of server connections, absorbs client reconnections transparently during database restarts, and reduces the per-connection overhead that accumulates when many application replicas connect directly to PostgreSQL
